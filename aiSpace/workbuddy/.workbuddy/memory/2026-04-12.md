## B-05 structuredClone DataCloneError 修复（Phase 1 阶段）

**问题**：点击节点时报 `DataCloneError: Failed to execute 'structuredClone' on 'Window': #<Object> could not be cloned.`，发生在 FieldProperties.vue watch 回调中。

**根因**：Phase 1 Batch 2 引入 `structuredClone(schema["x-component-props"])`，但 `props.schema` 是 Vue reactive proxy，`schema["x-component-props"]` 也是 reactive proxy。`structuredClone` 无法克隆含有 Vue 私有状态的 proxy 对象。

**修复**：先用 `toRaw()` 脱掉 reactive 外衣，再 `structuredClone`：
```js
// 错误
const cp = structuredClone(schema["x-component-props"] ?? {});
// 正确
const cp = structuredClone(toRaw(schema)["x-component-props"] ?? {});
```

**副作用**：第二个报错 `emitsOptions is null` 是 B-05 导致的级联效应，B-05 修复后自动消除。

**验证**：vue-tsc --noEmit 0 errors，vitest 149/149 全部通过。

## B-04 "流式节点无法拖拽"回归 Bug 修复

**问题**：B-04 修复后流式节点无法拖拽排序。

**根因**：B-04 把 `nodeId !== selectedNodeId.value` 从 guard 条件里删掉了，导致 `isDirectClick=true` 时 guard 跳过，第一次点击就启动拖拽。配合"两次点击模式"的设计意图，需要这个条件来区分"点未选中节点=选中"和"点已选中节点=拖拽"。

**修复**：恢复 `nodeId !== selectedNodeId.value` 条件到 guard：
```js
if (!isAbsolute && nodeId !== selectedNodeId.value && !isDirectClick) {
  e.stopPropagation()
  return
}
```

## 修复（第八次，最终）：使用 composedPath 解决容器内子节点点击问题

**问题**：点击卡片容器内的"备注"字段时，每次都选中卡片容器，而不是备注字段。

**根因分析**：

之前的修复方案使用了 `setTimeout(0)` + `data-click-suppressed` 属性来阻止 click 事件，但这个方案有致命缺陷：

```
1. mousedown → setTimeout(0) 注册回调（清除属性）
2. click 事件触发
3. setTimeout 回调执行（清除属性）
```

由于 `setTimeout(0)` 是在下一个 event loop task 中执行，而 click 事件在 mouseup 后立即触发（也是当前 event loop），两者执行顺序不确定。

**正确修复**：使用 `e.composedPath()` 在 `VoidContainer.handleClick` 中检查事件是否来自内部子节点

```typescript
// VoidContainer.vue
function handleClick(e: MouseEvent): void {
  // 检查事件路径：如果 click 来自 VoidContainer 内部的子节点（有 data-field-id），
  // 说明子节点会处理选中，这里应该让事件冒泡，不重复选中容器
  const composedPath = e.composedPath()
  const clickedOnChildNode = composedPath.some(el => {
    if (el instanceof HTMLElement && el.hasAttribute?.('data-field-id')) {
      const fieldId = el.getAttribute('data-field-id')
      // 如果找到的是容器自身，跳过
      if (fieldId === props.schema["x-id"]) return false
      // 如果找到的是子节点（FieldRenderer/VoidContainer），返回 true
      return true
    }
    return false
  })
  
  // 如果点击的是内部子节点，让子节点处理选中，容器不处理
  if (clickedOnChildNode) {
    return
  }
  // ... 正常选中逻辑
}
```

**事件流程（修复后）**：
```
1. mousedown on 备注 label
   └── CanvasOverlay.handleMouseDown → return (不启动拖拽)

2. click 事件
   └── FieldRenderer.handleWrapperClick → 选中备注字段 ✅
   └── VoidContainer.handleClick → composedPath 检查
       ├── 发现是子节点 → return ✅
```

**涉及文件**：
- `prototype/src/designer/composables/useDragInteraction.ts`（移除无效的 setTimeout）
- `prototype/src/renderer/VoidContainer.vue`（使用 composedPath 过滤）

**验证**：
- vue-tsc --noEmit ✅
- vitest run 149/149 ✅

## 关键教训

**事件机制**：
- `composedPath()` 返回事件经过的所有元素（包括 shadow DOM）
- 用于判断事件真正来源，比 `closest()` 更精确
- `setTimeout(0)` 不适合用于协调 DOM 事件时序
